# 文档规范

## 1. 存放位置与范围

- **长期文档必须进入 `docs/`**（会被长期维护/持续引用的内容）。
- **临时文档必须进入 `temp/`**（一次性、短期有效、尚未定稿的记录），不要放进 `docs/`。
- 临时文档如果被证明有长期价值，应在合并/发布前提炼为长期文档并迁入 `docs/`。

## 2. 目录规划

推荐结构（可按项目扩展）：

```text
docs/
  文档索引.md                  # 文档索引（入口）
  architecture/                # 架构、分层、模块边界、数据流
  standards/                   # 规范（代码、注释、文档、提交等）
  adr/                         # 架构决策记录（ADR）
  runbook/                     # 运维/发布/排障手册（如需要）
```

## 3. 命名与写作要求

- `docs/` 下的文档 **必须为中文文档**（以中文为主；允许在括号中补充英文术语/缩写）。
- `docs/` 下的文档 **文件名必须为中文**（建议使用“中文短语.md”，不使用空格；必要时可使用 `-` 连接）。
- 每篇长期文档至少包含：
  - 背景/问题
  - 结论/决策
  - 约束与边界（适用范围、非目标）
  - 示例（能复制粘贴或能落到代码结构）
- 任何会影响团队协作的规则/流程，必须写进 `docs/standards/`。

## 4. 变更流程

- 文档与代码同仓同审：规则变更必须随 PR 一起修改文档。
- 重大架构调整建议写 ADR：
  - 在 `docs/adr/` 新增一篇，记录“为什么这么做”和“替代方案为什么不选”。
